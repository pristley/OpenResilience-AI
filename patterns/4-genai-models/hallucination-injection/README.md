# Pattern: Hallucination Injection & Recovery

> When LLMs generate false information, can your system detect it?

## Problem Statement

LLMs occasionally generate factually incorrect outputs ("hallucinations")—outputs that are coherent, plausible, and confident but factually wrong. Examples:

- **False citations**: "According to the 2023 Smith Report..." (report doesn't exist or says something different)
- **Incorrect facts**: "Python was created in 1990" (actually 1989)
- **Fake statistics**: "90% of developers prefer X" (no source for this stat)
- **Confabulated details**: Inventing API endpoints, function names, or system behaviors that don't exist
- **Contradictions**: Claiming "Service A is down" while also saying "Service A is healthy" in the same response

Your system must detect hallucinations, validate them against ground truth, and gracefully degrade when they occur.

---

## Why It Matters

- **Impact metric**: Wrong information served to users (compliance violations, customer harm, reputational damage)
- **Detection latency**: Often invisible until a user complains or downstream validation fails
- **Blast radius**: 
  - *Advisory use* (chatbot): User sees wrong info but can verify
  - *Binding use* (auto-generated code, SQL, infrastructure configs): Hallucination executed → system breaks
  - *Training feedback loop*: User "corrects" hallucination → model retrains on bad data → worse hallucinations
- **Cost**: Misinformation, manual review overhead, regulatory penalties

---

## How It Fails

### Mechanism

1. **LLM generates content** based on training data + prompt + sampling (temperature > 0)
2. **Model is confident** even though it's wrong (high probability assigned to hallucinated tokens)
3. **Application doesn't validate** — either trusts the LLM or doesn't have ground truth to check against
4. **User sees hallucination** or it's passed to downstream system (code execution, RAG retrieval, API call)
5. **Feedback loop amplifies**: User corrects hallucination → model sees "correction" as training signal → learns wrong fact

### Observable Signals

- **Metric**: `hallucination_rate` = (outputs with factual errors) / (total outputs) — hard to measure in real-time
- **Metric**: `confidence_factuality_divergence` = abs(model_confidence - actual_correctness) — spikes when hallucinating
- **Log**: Citation validation failures — cited source doesn't support claim
- **Log**: Constraint violations — output violates known schema/business rules
- **Trace**: Retrieval source irrelevance — RAG retriever returned low-relevance documents
- **Trace**: Token probability collapse — model's next-token probabilities become uniform (confusion)
- **User signal**: Contradiction feedback or explicit "this is wrong" reports

### Time to Detect

- **Best case** (validation in place): 100ms (real-time constraint checker catches it)
- **Realistic** (sampling + human review): 5-30 minutes (QA pipeline + manual spot-check)
- **Worst case** (user reports): Hours to days (customer discovers it)

### Blast Radius

- **Direct**: User sees wrong information (trust erosion, wrong decision)
- **Downstream**:
  - Hallucination fed into recommendation engine → bad recommendations cascade
  - Code generated by LLM executed → system error/security risk
  - SQL query hallucinated → wrong data retrieved
  - API call to non-existent endpoint → failure cascade
- **Training**: Bad feedback creates new hallucinations (positive feedback loop)
- **Reputation**: Public-facing errors = brand damage

## Resilience Strategy

### Prevention

How to reduce hallucinations before they happen:

1. **Prompt engineering** (essential)
   - Add fact-checking constraints: "If you're unsure, say 'I don't know' instead of guessing"
   - Request citations: "Provide sources for each claim"
   - Use structured output: "Return JSON with `fact`, `source_url`, `confidence`"
   - Why it helps: Reduces model's tendency to fill gaps
   - Trade-off: May increase latency (longer prompt), reduce coverage (more "I don't know" responses)

2. **Retrieval Augmentation (RAG)** (essential for fact-heavy tasks)
   - Ground LLM responses in trusted sources (docs, DB, knowledge base)
   - Validate retrieval relevance before passing to LLM (cosine similarity > 0.8)
   - Why it helps: LLM has facts to cite, reduces confabulation
   - Trade-off: Latency + complexity; garbage in = garbage out if retrieval is bad

3. **Confidence thresholding** (important)
   - Only serve outputs where `confidence >= 0.8`
   - Return "I'm not confident enough to answer" for low-confidence
   - Why it helps: Filters out most hallucinations
   - Trade-off: Some true but uncertain statements blocked; reduced coverage

4. **Constraint validation** (critical for binding use cases)
   - Check output against schema (does generated code have valid syntax?)
   - Validate against business rules (is generated price in valid range?)
   - Semantic validation (does output contradict other facts in system?)
   - Why it helps: Catches hallucinations before they cause harm
   - Trade-off: Need to define all constraints up front

5. **Temperature & sampling control** (tuning)
   - Lower temperature (0.3-0.5) = more deterministic, fewer hallucinations
   - Higher temperature (0.8+) = more creative, more hallucinations
   - Use temperature=0 for fact-based tasks, temperature>0 for creative tasks
   - Why it helps: Reduces variance
   - Trade-off: Lower creativity; higher token cost if needing more attempts

6. **Model selection** (strategic)
   - Larger models hallucinate less (GPT-4 << GPT-3.5)
   - Domain-specific models (legal LLM, medical LLM) hallucinate less
   - Constitution-trained models (Claude) have lower hallucination rates
   - Why it helps: Foundation quality matters
   - Trade-off: Cost + latency

### Detection

How to know hallucinations are happening:

**Real-time validation:**

```yaml
facts:
  - metric_name: hallucination_confidence_divergence
    definition: abs(model_predicted_confidence - actual_factuality_score)
    alert_threshold: > 0.3
    calculation: Compare LLM's confidence vs. ground-truth validation

  - metric_name: citation_validation_failure_rate
    definition: (claims with unsupported citations) / (total claims)
    alert_threshold: > 0.05  # > 5% of citations invalid
    calculation: Check if cited sources actually support the claim

  - metric_name: constraint_violation_rate
    definition: (outputs violating schema/rules) / (total outputs)
    alert_threshold: > 0.01
    calculation: Validate output against defined constraints

logs:
  - "Citation validation failed: LLM cited [source] but [source] contradicts claim"
  - "Constraint violation: Generated price $X outside allowed range [$min, $max]"
  - "Semantic contradiction: LLM claims 'X is true' but earlier said 'X is false'"
  - "Retrieval irrelevance: RAG returned docs with similarity < 0.7, LLM may confabulate"

traces:
  - retrieval_source_relevance (cosine similarity of retrieved docs)
  - token_probability_entropy (uniform distribution = confusion)
  - latency_spike (model struggling = more hallucinations)
```

**Observability checklist:**

- [ ] **Metric**: Citation validation pass/fail rate
- [ ] **Metric**: Confidence vs. factuality divergence
- [ ] **Metric**: Constraint violation rate
- [ ] **Log**: Validation failures with claim + source + actual fact
- [ ] **Trace**: Retrieval relevance scores
- [ ] **Health check**: Can validator reach ground-truth sources?

### Recovery

How to handle hallucinations when they occur:

**Automatic:**
- Circuit breaker on hallucination rate: If > 5% of outputs hallucinate, switch to fallback mode
- Graceful degrade: Return "I don't know" instead of hallucinating
- Fallback model: Switch to more conservative model (lower temperature, smaller, Constitution-trained)
- Retry with different sampling: Sometimes re-generating helps (different hallucination)

**Manual intervention:**
1. **Identify the hallucination**: Query ground-truth sources for facts that should be known
2. **Assess impact**: Did users see it? Was it executed? Did it feed into training?
3. **Rollback if needed**: Revert to prior LLM version or remove poisoned training data
4. **Add constraint**: If hallucination is systematic, add validation rule to catch it next time
5. **Update prompt**: If retrieval failed, add retrieval quality check to prompt

## Chaos Experiment: Hallucination Injection & Recovery

Test whether your system detects and gracefully handles hallucinations.

### Prerequisites

- Access to LLM (OpenAI, Anthropic, or local)
- Ground-truth source (database, web docs, knowledge base)
- Validation system in place
- Monitoring dashboards set up

### Running the Experiment

```python
# experiments/genai-models/hallucination-injection/run.py
"""
Chaos experiment: Inject hallucinations and verify system resilience
"""
import re
import random
from typing import Optional
import anthropic
import requests

def inject_hallucination(response: str, fact_type: str = "date") -> str:
    """Replace a fact in LLM output with plausible falsehood."""
    
    if fact_type == "date":
        # Replace years (2024 -> 2025, 1989 -> 1988)
        return re.sub(r'(19|20)\d{2}', 
                     lambda m: str(int(m.group()) + random.randint(-2, 2)), 
                     response)
    
    elif fact_type == "number":
        # Replace percentages with random wrong ones
        return re.sub(r'(\d+)%', 
                     lambda m: f"{random.randint(1, 99)}%", 
                     response)
    
    elif fact_type == "entity":
        # Replace company names with fake ones
        companies = ["Acme Corp", "TechFlow Inc", "DataVault Ltd"]
        return re.sub(r'(Apple|Google|Microsoft|Amazon)',
                     lambda m: random.choice(companies),
                     response)
    
    elif fact_type == "citation":
        # Insert fake citation
        fake_cites = [
            "2023 Smith et al. Research Report",
            "Annual Tech Trends 2024",
            "Industry Analysis Q2 2024"
        ]
        return response + f"\n\nSource: {random.choice(fake_cites)}"
    
    return response

def validate_against_ground_truth(claim: str, sources: dict) -> dict:
    """Check if claim is supported by ground truth sources."""
    
    results = {
        "claim": claim,
        "valid": False,
        "supporting_sources": [],
        "contradicting_sources": []
    }
    
    for source_name, source_text in sources.items():
        if any(fact in source_text for fact in claim.split()):
            results["supporting_sources"].append(source_name)
            results["valid"] = True
        else:
            results["contradicting_sources"].append(source_name)
    
    return results

def test_hallucination_injection():
    """
    1. Ask LLM a factual question
    2. Inject hallucination
    3. Verify system detects it
    4. Verify graceful degradation
    """
    
    print("🧪 Hallucination Injection Experiment")
    print("=" * 60)
    
    client = anthropic.Anthropic()
    
    # Ground truth
    ground_truth = {
        "python_history": "Python was created by Guido van Rossum and released in 1989",
        "python_2024": "Python is one of the most popular programming languages in 2024",
        "python_typing": "Python 3.5+ supports type hints"
    }
    
    # Test prompt
    prompt = """Who created Python and when was it released? 
    Please provide a concise answer with the creator's name and year."""
    
    # 1. Get real response
    print("\n[Phase 1] Getting baseline LLM response...")
    real_message = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=200,
        messages=[{"role": "user", "content": prompt}]
    )
    real_response = real_message.content[0].text
    print(f"Real response: {real_response[:100]}...")
    
    # 2. Inject hallucination
    print("\n[Phase 2] Injecting hallucination (wrong date)...")
    injected_response = inject_hallucination(real_response, "date")
    print(f"Injected response: {injected_response[:100]}...")
    
    # 3. Validate real vs. injected
    print("\n[Phase 3] Validating responses against ground truth...")
    
    real_validation = validate_against_ground_truth(
        real_response,
        ground_truth
    )
    print(f"Real response validation:")
    print(f"  - Valid: {real_validation['valid']}")
    print(f"  - Supporting sources: {real_validation['supporting_sources']}")
    
    injected_validation = validate_against_ground_truth(
        injected_response,
        ground_truth
    )
    print(f"Injected response validation:")
    print(f"  - Valid: {injected_validation['valid']}")
    print(f"  - Contradicting sources: {injected_validation['contradicting_sources']}")
    
    # 4. Test: System should flag injected hallucination
    print("\n[Phase 4] Verifying hallucination detection...")
    assert real_validation['valid'] == True, "Real response should be valid"
    assert injected_validation['valid'] == False, "Injected hallucination should be flagged as invalid"
    print("✅ Hallucination correctly identified")
    
    # 5. Test: System should degrade gracefully
    print("\n[Phase 5] Testing graceful degradation...")
    
    # Simulate fallback: If confidence < 0.7, return "I don't know"
    fallback_threshold = 0.7
    
    # For testing, assume injected response has low confidence score
    injected_confidence = 0.5  # Simulated
    
    if injected_confidence < fallback_threshold:
        response_to_serve = "I don't have high confidence in my answer to that question."
        print(f"Fallback activated: '{response_to_serve}'")
        print("✅ System gracefully degraded instead of serving hallucination")
    else:
        print("❌ System would serve hallucinated response")
        raise AssertionError("System did not degrade gracefully")
    
    # 6. Summary
    print("\n" + "=" * 60)
    print("✅ EXPERIMENT PASSED")
    print("=" * 60)
    print(f"Real response valid: {real_validation['valid']}")
    print(f"Injected response detected as invalid: {not injected_validation['valid']}")
    print(f"System can degrade gracefully: True")

if __name__ == "__main__":
    test_hallucination_injection()
```

Watch these metrics:

- `validation_latency_ms` — Time to validate facts (should be < 500ms)
- `confidence_factuality_correlation` — Are high-confidence outputs actually correct?
- `fallback_activation_rate` — How often does system return "I don't know"?

### Expected Outcome

✅ **Success criteria:**
- Injected hallucinations detected within 100ms (real-time)
- System serves fallback instead of hallucination
- No hallucinated output reaches end users
- Validation catches 90%+ of injected errors

❌ **Failure signs:**
- Hallucinations served directly to users
- Citation validation missing (cited sources not checked)
- No fallback mechanism (always serves LLM output)
- Validation disabled in production

## Lessons Learned

### Real-World Incidents

**Incident 1: Support Chatbot Misconfiguration**
- **Root cause**: RAG retriever returned irrelevant docs; LLM filled gaps with hallucination
- **Hallucination**: Chatbot told customer to "restart the server" when docs meant "restart the service"
- **Impact**: 50+ customers attempted wrong fix; server downtime
- **Detection**: 2 hours (customer escalation)
- **Fix**: Added semantic relevance check (cosine similarity > 0.8) + citation validation
- **Prevention**: Now require validation pass before serving LLM output

**Incident 2: Code Generation Execution**
- **Root cause**: LLM-generated SQL had wrong table names (hallucinated)
- **Hallucination**: Generated query referenced non-existent columns
- **Impact**: Auto-executed query failed; data pipeline broke; data loss investigation
- **Detection**: 10 minutes (monitoring alert on query failure rate)
- **Fix**: Added SQL syntax + schema validation before execution
- **Prevention**: For binding outputs (code, SQL, configs), always validate before execution

**Incident 3: Fine-tuned Model Regression**
- **Root cause**: Model retrained on user-submitted "corrections" that were actually wrong
- **Hallucination**: Model started generating the same wrong facts systematically
- **Impact**: Accuracy dropped from 92% to 78%
- **Detection**: 3 days (weekly accuracy report)
- **Fix**: Added human review step for training data; reverted to prior model
- **Prevention**: Never automatically train on user feedback; require QA approval first

### Key Takeaways

- **Confidence ≠ Correctness**: High-probability outputs can still be hallucinations
- **Retrieval quality matters**: Bad RAG sources → guaranteed hallucinations
- **Validate everything**: Especially code, SQL, API calls, and binding decisions
- **Graceful degradation**: "I don't know" >> hallucinated wrong answer
- **Feedback loops amplify**: Bad correction + retraining = systemic hallucinations
- **Transparency helps**: "Source: [URL]" makes hallucinations obvious to users

### Anti-Patterns

- ❌ **"LLM has high confidence, so it must be right"** → Confidence ≠ correctness
- ❌ **"We don't have time to validate every output"** → Hallucinations cost more time to fix
- ❌ **"Just lower the temperature"** → Reduces creativity; doesn't eliminate hallucinations
- ❌ **"Auto-train on user corrections"** → Creates positive feedback loop for bad data
- ❌ **"Citation required by prompt, so must be true"** → LLM can hallucinate citations too
